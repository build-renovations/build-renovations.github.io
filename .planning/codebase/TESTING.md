# Testing Patterns

## Testing model

This repository does not use a unit-test or integration-test framework. There are no `test`, `spec`, or browser-automation source suites in the site code. Quality is enforced through build checks, shell scripts, and generated audit artifacts.

## Primary verification entry points

- `scripts/qa.sh` is the baseline validation command.
- `scripts/gates.sh` layers stricter phase-gate checks on top of the base build.
- `scripts/serve.sh` is the local preview path and uses the same Ruby compatibility preload as the build script.

## What `scripts/qa.sh` verifies

`scripts/qa.sh` is the closest thing to a core test suite. It performs four kinds of checks:

1. Image policy enforcement for `assets/images/`
   - rejects `.jpg`, `.jpeg`, `.png`, `.tif`, and `.tiff`
   - uses `ffprobe` to fail on metadata inside `.webp`
   - scans `.svg` for metadata-like XML blocks
2. Clean rebuild of `_site/`
   - removes existing generated output before building
3. Static build validation
   - runs `bundle exec jekyll build` with `scripts/jekyll_compat.rb` preloaded
4. Smoke assertions on expected outputs
   - `_site/index.html`
   - `_site/en/index.html`
   - `_site/services/index.html`
   - `_site/en/services/index.html`
   - `_site/sitemap.xml`

The main implication: a change is not considered safe if it only "looks fine" in source. It must still build into the expected bilingual output set.

## What `scripts/gates.sh` adds

`scripts/gates.sh` acts like a release gate runner, not a developer unit-test runner. It codifies staged acceptance:

- `gate1`
  - requires strategy artifacts under `agents/seo/` and `agents/design/`
- `gate2`
  - requires design, content, and front-matter artifacts
- `gate3`
  - reruns `scripts/qa.sh`
  - requires `agents/status/dev.md`
  - requires `Status: DONE` inside `agents/status/dev.md`
- `gate4`
  - reruns `scripts/qa.sh`
  - serves `_site/` locally with `python3 -m http.server`
  - runs Lighthouse via `npx --yes lighthouse`
  - runs accessibility checks via `npx --yes pa11y`
  - runs broken-link crawling via `npx --yes linkinator`

This means testing is split between source validation and rendered-site validation. The rendered HTML in `_site/` is the thing being audited.

## Current evidence and artifacts

- Gate status is summarized in `agents/status/gates.md`.
- Gate 4 evidence lives in `agents/status/reports/`.
- Existing artifacts include:
  - `agents/status/reports/gate4-summary.md`
  - `agents/status/reports/lighthouse-home.json`
  - `agents/status/reports/lighthouse-services.json`
  - `agents/status/reports/lighthouse-about.json`

These files show the repo’s testing style clearly: audit outputs are committed as evidence, not regenerated by an application test harness.

## Testing patterns contributors should follow

- Run `./scripts/qa.sh` after any change that touches layouts, includes, data, content structure, assets, or configuration.
- Run the relevant gate command when a change affects readiness criteria, usually `./scripts/gates.sh gate3` or `./scripts/gates.sh gate4`.
- Treat bilingual integrity as part of testing:
  - verify both `/` and `/en/` build
  - verify translated pages keep matching `translation_key`
  - verify language switching, canonical links, and `hreflang` generation still work through `_includes/head.html` and `_includes/header.html`
- For layout or navigation changes, inspect generated files in `_site/` instead of trusting source templates alone.
- For image additions, expect `scripts/qa.sh` to fail if the asset format or metadata policy is violated.

## Important gaps

- No unit tests cover `assets/js/site.js`. JS behavior is validated indirectly through browser-based audits and manual QA.
- No snapshot tests exist for Liquid output from `_layouts/` or `_includes/`.
- No schema validation exists for `_data/*.yml`; malformed YAML is caught only when Jekyll build fails or rendered pages break.
- Gate 4 samples only selected pages for Lighthouse and accessibility checks, not every route in both languages.
- There is no dedicated regression test for every `translation_key` pair. That contract is enforced mainly by convention and rendered output review.

## Practical test strategy for new work

- Source-level change:
  - update both languages and shared data where applicable
  - confirm front matter still contains the required bilingual metadata
- Build-level validation:
  - run `./scripts/qa.sh`
- Release-level validation for significant UI/SEO changes:
  - run `./scripts/gates.sh gate4`
- Evidence review:
  - check `agents/status/gates.md`
  - inspect the relevant JSON or summary files in `agents/status/reports/`

In this repo, "tested" means "builds cleanly, generates the expected bilingual site, and passes the scripted rendered-page audits."
