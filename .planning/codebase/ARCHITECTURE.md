# Repository Architecture

## Scope

This repository combines a deployable bilingual GitHub Pages site with a non-deployable multi-agent orchestration workspace.

- Published site: Jekyll source files in the repository root plus `_config.yml`, `_data/`, `_includes/`, `_layouts/`, `assets/`, content directories such as `services/`, `about/`, `process/`, `faq/`, `contact/`, and their English mirrors under `en/`.
- Non-published workspace: `agents/`, `docs/`, `.codex/`, and `.planning/`.
- Build output: `_site/` generated by Jekyll and treated as disposable.

MCP was not needed for this mapping pass. The repository already exposes the required architectural evidence locally.

## Primary Runtime Model

The site is a static Jekyll application configured for GitHub Pages safe mode in `_config.yml`.

- `theme: null` indicates a fully custom site rather than a packaged theme.
- The plugin set is limited to `jekyll-seo-tag` and `jekyll-sitemap` in `_config.yml`, which matches GitHub Pages-safe expectations.
- `permalink: pretty` drives directory-style URLs for content pages.
- `exclude:` in `_config.yml` explicitly keeps operational material out of the published site, including `agents/`, `docs/`, `scripts/`, `AGENTS.md`, and dependency directories.

The actual deployment model is root-published Ukrainian content and mirrored English content under `/en/`.

- Ukrainian default entrypoint: `index.md`
- English homepage: `en/index.md`
- Ukrainian section roots: `about/index.md`, `contact/index.md`, `faq/index.md`, `process/index.md`, `services/index.md`
- English mirrors: `en/about/index.md`, `en/contact/index.md`, `en/faq/index.md`, `en/process/index.md`, `en/services/index.md`

## Content Composition Layers

The architecture is intentionally layered so content, templates, and presentation can evolve independently.

### 1. Page Front Matter Layer

Markdown pages mostly provide routing, metadata, and page identity rather than large amounts of bespoke markup.

- `lang` defines the active locale.
- `translation_key` groups translated siblings.
- `layout` selects rendering behavior such as `home`, `page`, `process`, or `service_detail`.
- `service_key` on service detail pages links a page to a structured record in `_data/service_pages.yml`.
- `seo_title` and `seo_description` override defaults when needed.

This pattern is visible in files such as `index.md`, `process/index.md`, `services/plumbing/index.md`, `en/index.md`, and `en/services/plumbing/index.md`.

### 2. Shared Data Layer

Repeated site content is centralized in `_data/` rather than duplicated across templates.

- Navigation: `_data/navigation.yml`
- UI strings and bilingual microcopy: `_data/translations.yml`
- Service-card content for listing sections: `_data/services.yml`
- Service-detail content model: `_data/service_pages.yml`
- Process list for homepage: `_data/process.yml`
- Process landing-page content model: `_data/process_page.yml`
- Trust/proof cards: `_data/proof.yml`
- FAQ content: `_data/faqs.yml`
- Image-led gallery and storytelling blocks: `_data/work_examples.yml`

This is the core maintainability strategy of the repository. Layouts render from data files keyed by language or service identifier, while pages stay thin.

### 3. Template Layer

Reusable Liquid templates define page families.

- Base document shell: `_layouts/default.html`
- Homepage composition: `_layouts/home.html`
- Generic content page: `_layouts/page.html`
- Process page template: `_layouts/process.html`
- Service detail template: `_layouts/service_detail.html`

Shared chrome and metadata are implemented through includes.

- SEO, canonical, hreflang, and JSON-LD: `_includes/head.html`
- Navigation and language switcher: `_includes/header.html`
- Footer: `_includes/footer.html`
- Shared portfolio/gallery section: `_includes/work-examples.html`

### 4. Presentation Layer

The front end is intentionally lightweight.

- Global styles and component styling live in `assets/css/site.css`.
- Small interaction layer lives in `assets/js/site.js`.

JavaScript is limited to progressive enhancement:

- mobile navigation toggle
- scroll progress indicator
- pointer-driven tilt effect on selected cards
- intersection-observer reveal animations

There is no heavy client-side framework, no bundler, and no SPA routing. This matches the static-site and GitHub Pages constraints described in `README.md`, `AGENTS.md`, and `docs/agents/development.md`.

## Bilingual Routing Architecture

The multilingual model is implemented without translation plugins.

- Root paths are Ukrainian, for example `services/index.md` with permalink `/services/`.
- English pages are duplicated structurally under `en/`, for example `en/services/index.md` with permalink `/en/services/`.
- Language affinity is maintained by matching `translation_key` values across sibling pages.

Two templates consume that contract directly:

- `_includes/head.html` scans `site.pages` by `translation_key` to emit canonical and `hreflang` alternates.
- `_includes/header.html` scans `site.pages` by `translation_key` to build the UA/EN language switcher.

This is a critical architectural decision: language switching and SEO alternates depend on page front matter consistency, not on a plugin or external registry.

## Service and Gallery Architecture

The service area is modeled as two related systems.

### Service Overview System

`_data/services.yml` powers list-style service cards rendered on the homepage in `_layouts/home.html`.

This dataset provides:

- `key`
- `url`
- `image`
- bilingual titles and descriptions

### Service Detail System

Detailed pages use `service_key` in the page front matter and resolve content through `_data/service_pages.yml` in `_layouts/service_detail.html`.

This yields:

- hero content
- summary copy
- service scope bullets
- related service references
- shared image source per service family

### Work Examples System

`_includes/work-examples.html` reads `_data/work_examples.yml` and can operate in two modes:

- full-width section mode on the homepage
- compact embedded mode inside listing/detail pages

It also supports `filter_key`, which allows specific service detail pages to show only matching gallery content. This is the main reusable rich-content component in the site.

## SEO and Metadata Architecture

Metadata logic is centralized in `_includes/head.html`.

The include computes:

- page title fallback rules
- page description fallback rules
- canonical URL from `page.url | absolute_url`
- Open Graph and Twitter metadata
- noindex behavior from `site.seo.allow_indexing`
- `hreflang` alternates and `x-default`
- JSON-LD business schema using `site.company` and `site.seo`

Repository-level metadata inputs come from:

- `_config.yml` for `site.seo`, `site.company`, and defaults
- page front matter for per-page overrides
- `translation_key` and `lang` for alternate-language generation

`robots.txt` is also a Jekyll-rendered template and follows the same indexing flag in `_config.yml`.

## Build and Verification Architecture

The repository uses small shell scripts rather than a task runner.

- `scripts/serve.sh` clears `_site/` and runs local Jekyll serve.
- `scripts/qa.sh` enforces image policy, runs Jekyll build, and asserts key output files exist.
- `scripts/gates.sh` layers gate checks on top of `scripts/qa.sh` and optionally runs Lighthouse, Pa11y, and Linkinator.
- `scripts/jekyll_compat.rb` patches Ruby compatibility for the older GitHub Pages Jekyll stack.

The dependency contract is intentionally conservative.

- `Gemfile` uses `github-pages` instead of custom plugin drift.
- `README.md` documents that local scripts preload the compatibility shim because the Pages gem still depends on Jekyll 3.x behavior.

This means the repository architecture optimizes for deployability and reproducibility over modern Jekyll customization.

## Architectural Strengths

- Strong separation between deployable site inputs and orchestration/process files via `_config.yml` excludes.
- Lightweight multilingual approach that stays GitHub Pages compatible.
- Data-driven content architecture in `_data/` reduces duplication across UA and EN.
- Layout specialization is proportionate to the site size: only four layout families cover the published pages.
- Minimal JavaScript keeps the runtime surface small.
- Verification scripts encode both content-policy checks and deployability checks.

## Architectural Constraints and Coupling Points

- Translation integrity depends on manually keeping `translation_key`, `lang`, and permalinks aligned across paired pages.
- Service pages depend on `service_key` values matching keys in `_data/service_pages.yml`; there is no schema enforcement beyond build-time Liquid behavior.
- Homepage and gallery sections are tightly coupled to `_data/` file shapes, especially `_data/work_examples.yml` and `_data/services.yml`.
- `site.pages` based language discovery in `_includes/head.html` and `_includes/header.html` assumes all translated pages are regular pages, not collection documents.
- The repository carries operational screenshots and review images in the root, which are not clearly partitioned from deployable inputs unless readers understand the exclude rules and top-level file intent.

## Practical Mental Model

Treat the repository as two adjacent systems sharing one Git root:

1. A static bilingual Jekyll site built from root content, `_data/`, layouts, includes, assets, and a conservative GitHub Pages gem stack.
2. A non-published orchestration framework that plans, verifies, and documents work around that site through `docs/`, `agents/`, `.codex/`, and helper scripts.

Most user-facing behavior comes from changing data records and front matter, not from adding new rendering code. Most process rigor comes from scripts and orchestration docs, not from the site runtime itself.
